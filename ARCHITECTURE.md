# Architecture

agentsync is a small, dependency-light CLI. This document explains how the
pieces fit together and, in particular, how to add a new target adapter.

## High-level flow

```
agentsync.config.yaml ──┐
                         ├─► loadConfig()  ─► AgentSyncConfig
rules/*.md ──────────────┘
                         │
                         ├─► loadRules()   ─► RuleFragment[]
                         │
                         ├─► resolveTargets() ─► ResolvedTarget[]  (config + adapter registry)
                         │
                         └─► adapter.render({ config, rules }) ─► generated file content (string)
```

Every command (`build`, `check`, `diff`) shares this same pipeline via
`src/commands/shared.ts`'s `prepareRun()`. They differ only in what they do
with the rendered strings:

- **build** (`src/commands/build.ts`) writes each rendered string to its
  output path.
- **check** (`src/commands/check.ts`) re-renders in memory and does an exact
  string comparison against the file already on disk. If a file is missing
  or its content differs, that target is reported as out of sync and the
  command exits `1`.
- **diff** (`src/commands/diff.ts`) does the same comparison as `check`, but
  additionally runs `src/core/diff.ts`'s line-based diff and prints a
  unified-style report instead of just yes/no.

### Why stateless comparison instead of a hash/lock file?

An earlier design considered writing a `.agentsync/state.json` manifest
(content hashes at last `build`) and having `check` compare against that
manifest. We deliberately didn't: a manifest can drift from the *actual*
generator logic (e.g. after upgrading agentsync itself, or after editing an
adapter), silently reporting "in sync" when the file on disk no longer
matches what the current code would produce. Re-rendering in memory on every
`check`/`diff` call is cheap (rule files are small Markdown documents) and
is always correct relative to the code actually running — the same
philosophy as `terraform plan` or `prettier --check`.

## Core modules (`src/core/`)

| Module | Responsibility |
| --- | --- |
| `types.ts` | Shared types: `AgentSyncConfig`, `RuleFragment`, `TargetAdapter`, etc. |
| `config.ts` | Loads and validates `agentsync.config.yaml`. No adapter knowledge. |
| `rules.ts` | Loads and orders Markdown fragments from the rules directory; splits a leading `# Heading` into a title. |
| `targets.ts` | Resolves which targets to process (`resolveTargets`), given config and an optional `--only` filter, by looking targets up in the adapter registry. |
| `diff.ts` | Dependency-free line-based diff (`diffLines`/`formatDiff`) used by `agentsync diff`. |
| `errors.ts` | `ConfigError`, `RulesError`, `TargetError` — typed errors with CLI-friendly messages. |

None of these modules know about specific target formats (Claude, Cursor,
...) — that knowledge lives entirely in `src/adapters/`.

## The adapter pattern (`src/adapters/`)

A target adapter is any object implementing `TargetAdapter`:

```ts
export interface TargetAdapter {
  id: string;              // matches the key under `targets:` in config
  label: string;            // human-readable, used in CLI output
  defaultOutput: string;    // used when config omits `output:`
  render(ctx: AdapterContext): string; // ctx = { config, rules }
}
```

`render()` is a pure function: given the parsed config and ordered rule
fragments, it returns the full contents of the target file as a string. It
must not perform I/O — reading and writing files is the commands' job, not
the adapter's. This keeps adapters trivially unit-testable (see
`tests/unit/adapters.test.ts`) and keeps `check`/`diff` correct, since they
rely on being able to re-render on demand without side effects.

Three of the four built-in adapters (`claude`, `agents`, `gemini`) share
nearly all of their logic via `src/adapters/shared.ts#renderMarkdownBody` —
a project heading, an optional description, an HTML-comment "generated by
agentsync, do not edit" notice, and one `##` section per rule fragment. The
`cursor` adapter wraps the same shared body in the YAML frontmatter Cursor's
`.mdc` rule format requires (`description`, `globs`, `alwaysApply`).

### Adding a new target adapter

1. Create `src/adapters/<id>.ts` exporting a `TargetAdapter`. Reuse
   `renderMarkdownBody()` from `shared.ts` if a plain Markdown body (with an
   optional wrapper, like Cursor's frontmatter) is all you need.
2. Register it in `src/adapters/index.ts`'s `builtinAdapters` array.
3. Add a `targets.<id>` entry to the example configs in `examples/` and to
   the "Supported targets" table in `README.md` / `README.ja.md`.
4. Add unit tests asserting the rendered output in
   `tests/unit/adapters.test.ts`, following the existing pattern.
5. That's it — `build`, `check`, and `diff` all work automatically, because
   they only depend on the `TargetAdapter` interface via
   `resolveTargets()`/`getAdapter()`, never on a specific target by name.

This is the "adapter pattern" the project spec asks for: the target-format
list is closed only by what's registered in `src/adapters/index.ts`, not by
anything in `src/commands/` or `src/core/`.

## CLI layer (`src/cli.ts`)

`src/cli.ts` is a thin [commander](https://github.com/tj/commander.js)
wrapper: it parses `argv`, builds a `Logger` (`src/utils/logger.ts`), and
calls the corresponding `run*()` function from `src/commands/`. Every
`run*()` function takes plain data (`{ cwd, configPath?, only? }`) and a
`Logger`, and **returns an exit code** rather than calling `process.exit()`
directly — this is what lets `tests/integration/lifecycle.test.ts` exercise
the full `init → build → check → diff` flow in-process, quickly and without
spawning subprocesses. `tests/integration/cli-binary.test.ts` additionally
spawns the real compiled `dist/cli.js` to cover the parts unit tests can't
(shebang, commander argument parsing, actual process exit codes).

## Library entry point (`src/index.ts`)

Everything the CLI uses is re-exported from `src/index.ts`, so agentsync can
be driven programmatically (e.g. from a custom build script, or a future
adapter distributed as a separate package) without spawning a child
process.
